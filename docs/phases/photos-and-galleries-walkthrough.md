## Decisions surfaced during walkthrough

Append-only log of decisions and emergent design changes from this phase. Each entry needs to land in its home doc at phase close.

- **Visibility filter UI redesign ‚Äî tag-type pill-dropdowns + multi-select via checkboxes, popover menu not modal.** The V1 design (one auto-pill per group the user had posted to, ModalSheet single-select) created a noisy auto-generated row + the wrong interaction shape. Replaced 2026-06-04 with fixed tag-type pills (`Dog ‚ñæ`, `Person ‚ñæ`, `Place ‚ñæ`, `Community ‚ñæ`, `Meet ‚ñæ`) opening `.dropdown-menu` popovers with checkbox rows. Multi-select within type (OR), AND across types. Albums-as-saved-filter-sets becomes the natural V2 forward direction (¬ß12 open item ‚Äî "albums-as-tag-type"). ‚Üí `features/profiles.md`
- **Composer affordance moves to page-action slot.** ShareMomentBar in-panel was clunky alongside the new filter row. Retired from PostsTab; "Post" button (outline + CameraPlus icon) registers via `setPageAction` for the Posts tab ‚Äî mirrors the Communities pattern. AppNav Camera+ stays suppressed via `setPageAction(node, { suppressCreate: true })` so they don't double up. ‚Üí `features/profiles.md` (updated)
- **Highlights field renamed ‚Äî `PetProfile.photoGallery` ‚Üí `PetProfile.highlights`.** Initial scope kept the legacy name to save churn; reversed 2026-06-04 after Highlights also landed on `UserProfile`. The asymmetric naming (`photoGallery` for dogs, `highlights` for users) was confusing enough to be worth the mechanical rename. ~6 callsites updated + seed migration. ‚Üí `features/profiles.md` (updated)
- **Highlights = a tag/state on a photo, not a separate collection.** A first pass of the Edit Highlights modal added an inline "Add photos" picker (grid of unpinned album photos with `+` overlays). Removed 2026-06-04 ‚Äî adding to Highlights is an in-context action via the per-post three-dot menu. Conceptual model: Highlights is a property of a photo (pin/unpin), not a list the owner builds. The Edit modal stays for order + bulk unpin only. Aligns with the ¬ß12 V2 direction (albums-as-tag-type). ‚Üí `features/profiles.md` (updated)
- **Global post-detail surface ‚Äî `PostLightbox`, no `/posts/[id]` route.** Surfaced as a need when wiring the tag_pending notification (no way to see the parent post) AND when the user asked for a full-screen image experience. Built `contexts/PostDetailContext.tsx` ‚Äî `usePostDetail().openPost(postId, { collection? })` overlays the app with a photo-led lightbox: dark backdrop, photo dominant on left (~60% desktop, full-bleed-top on mobile), sidebar on right (or below on mobile) with author + caption + tags + reactions + comments + PostKebabMenu. When opened with a `collection`, ‚Üê / ‚Üí keys and on-screen prev/next traverse the album in lightbox mode; without a collection (e.g. from a tag_pending notification), single post with no cross-post nav. Multi-photo posts get sub-photo navigation + `N/M` counter. No `/posts/[id]` route ‚Äî comments aren't a primary platform thread for Doggo; the lightbox is sufficient and keeps "tap ‚Üí see ‚Üí close" consistent. PhotoGrid passes the visible posts as the collection so the lightbox can navigate the album. ‚Üí `features/profiles.md` (note as the canonical post-detail surface)
- **Stub actions now open a "Demo placeholder" toast.** Previously stub menu items just closed the sheet ‚Äî silently inert. Replaced 2026-06-04 with a shared `StubFeatureContext` (`contexts/StubFeatureContext.tsx`) that slides in a non-modal toast (top-right desktop / top mobile, ~6s auto-dismiss, √ó to dismiss) identifying the feature and what it's waiting on. Built on top of a new reusable `<Alert>` component with 4 variants (info / success / warning / error) ‚Äî also available for inline page-level callouts. Wired into PostKebabMenu's four stubs (Edit post / Delete post / Report / Block). Available app-wide via `useStubNotice()` for future stub actions. ‚Üí `implementation/design-system.md` (note as a stub-action pattern)
- **Kebab is a floating popover, not a modal.** Initial implementation wrapped the per-post kebab menu in a `ModalSheet` (full-screen sheet). Replaced 2026-06-04 with a `.dropdown-menu` popover anchored top-right of the kebab button, click-outside-to-close, right-aligned with `min-width: 200px / max-width: 280px / width: max-content` so labels never wrap. Reuses the standard project pattern (same as Follow / RSVP / Sort menus). Lighter feel for a short list of actions; works on mobile + desktop. Specificity-fix note in punch list ‚Äî the shared `.dropdown-menu` rule comes later in source order so variant overrides need double-class selectors (`.dropdown-menu.post-kebab-menu`). ‚Üí no feature-doc update needed
- **Per-post kebab is the unified action surface; long-press removed.** The earlier long-press menu on auto-album tiles (Pin / Hide) read as finicky. Replaced 2026-06-04 with an expanded `PostKebabMenu` that covers all per-post actions: Pin/Unpin to Highlights, Hide/Show in album, Edit post (caption + tags consolidated into one stub), Delete post; plus the existing Untag {Dog} / Report / Block {Author} for non-owner posts. **Context-aware Highlights destination:** menu reads `usePathname()` ‚Äî on `/dogs/[id]` for a viewer-owned dog, Pin targets that dog's Highlights and Hide-from-album becomes available; on any other surface (feed, owner profile), Pin targets the viewer's own Highlights and Hide is suppressed (no album in context). Replaces the previous one-row-per-destination model with a single state-toggling row. `AlbumTileLongPressMenu` deleted. ‚Üí `features/profiles.md` (updated)
- **"posts ‚âà photos" ‚Äî no dedicated Photos tab.** Posts and Photos are the same collection (every Post has 1‚Äì4 photos). Posts tab gets a List ‚áÑ Grid view toggle instead of a parallel Photos tab. ‚Üí `features/profiles.md` (updated)
- **Highlights strip lives on `/dogs/[id]`, NOT on profile-level PetSummaryCard.** Reinforces Dog Profile phase's "canonical dog page" thesis. Owner-level "all my dogs" aggregate explicitly NOT built. ‚Üí `features/profiles.md` (updated)
- **Empty-state composer CTA opens the picker without pre-tagging the specific dog.** `openComposer({ initialTagPicker: "dog" })` opens the dog tag picker ‚Äî pre-selecting THIS specific dog needs a `prefilledTags` field on `PostComposerContext` which is out of scope. One-tap fill from the picker is acceptable for V1. ‚Üí no feature-doc update needed
- **Untag is a per-viewer suppression filter, not a post mutation.** Prototype has no editable-post context, so "Untag B√°ra" persists per viewer via `useUntagStore` and feeds into `/dogs/[id]` auto-album filtering. The post's `Post.tags[]` itself is unchanged ‚Äî the dog tag chip still renders on post chrome until a real editable-post store lands. ‚Üí `features/profiles.md` (updated)
- **Pending-tag approval model: grandfathered-default + explicit `DEMO_PENDING_TAGS` carve-out.** Initial implementation used a 14-day cutoff on both queue + album, which produced auto-approval-by-aging ‚Äî semantically wrong. Replaced 2026-06-04: dog-tag-on-owned-dog pairs default to approved (grandfathered); pairs in `DEMO_PENDING_TAGS` (lib/usePendingTagsStore.ts) default to pending. Pending stays pending forever until owner approves or rejects. Demo curation is a one-line edit to the constant. ‚Üí `features/profiles.md` (updated)
- **Pending-tag approval surface = notifications, NOT a dedicated About-tab section.** Initial implementation rendered `PendingTagsSection` inline on `/profile?tab=about`. Matches no shipping social platform pattern (Instagram / Twitter / Facebook all use notifications for tag review). Refactored 2026-06-04: added `tag_pending` to `NotificationType`, seeded 2 entries in `mockNotifications` matching `DEMO_PENDING_TAGS`, added inline Approve / Reject footer to the notifications row (mirrors `connection_request`). Resolved notifications filter out via `usePendingTagsStore.getDecision`. `PendingTagsSection` deleted. ‚Üí `features/profiles.md` (updated)
- **Photos/Posts unified across owner profile + dog page.** Initial implementation diverged: dog page had separate "Photos" grid + "Posts about {Dog}" sections; owner profile had a single Posts tab with List‚áÑGrid toggle + tag-type filters. Unified 2026-06-04 around the owner-profile shape: new `components/posts/PostsCollectionView.tsx` carries filter pills + view toggle + list/grid render. Used by both `PostsTab` AND the dog page (which drops the separate Photos + Posts-about sections in favor of a single Posts section under Highlights). Dog page section header is "Posts" (was "Posts about {Dog}"). Highlights strip + Edit modal + `usePhotoAlbumOverrides` generalized from dog-only to subject-id (dog OR user). New `UserProfile.highlights?: string[]` field seeded for the four anchor personas. Forward direction logged: Highlights becomes one album in a dropdown switcher when ¬ß12's "albums-as-tag-type" V2 lands. ‚Üí `features/profiles.md` (updated)
- **Resolver split ‚Äî `getPostsByDog` (viewer-aware, `lib/dogPosts.ts`) vs `getDogPosts` (ungated, `lib/mockShelters.ts`).** The shelter-side ungated helper survives for `RecentWalkers` (where walker activity is a public-ish surface; gating would over-restrict). Auto-album surfaces use the new helper. ‚Üí no feature-doc update needed (architectural)
- **Right-click on album tile triggers the long-press menu (desktop convenience).** Touch gets press-and-hold; mouse users get right-click as the equivalent affordance. Click-handling is intercepted so a long-press doesn't ALSO open the post modal. ‚Üí no feature-doc update needed
- **Punch-list spawn: P75 ‚Äî panel chrome glassmorphism consistency.** Some PageColumn panels glass, most don't. Recommendation logged: go opaque app-wide. Not in scope for this phase. ‚Üí `phases/punch-list.md`
- **Section rhythm ‚Äî global tight-header + larger body-gap pattern.** Two parallel section surfaces (`.profile-tab-stack > section` and `.dog-profile-section`) had diverged on internal gap (4px vs 16px). Aligned 2026-06-04 to a single rhythm rule that both classes share (visual chrome ‚Äî padding/background/border ‚Äî stays per-class because they render in different layout contexts): `gap: var(--space-sm)` (tight header ‚Üí first body item) + `> :nth-child(n+3) { margin-top: calc(--space-md - --space-sm) }` (larger gap between body items). CSS-only via a comma-separated selector list ‚Äî no JSX wrappers required. Future section surfaces can join by adding to the list. Also tightened `.dog-profile-section--header-only` padding-bottom (Posts about {Dog}) from `--space-lg` ‚Üí `--space-sm` so the feed-card's own top padding doesn't stack with the section's bottom padding into an oversized gap. ‚Üí `implementation/design-system.md` (note as a recurring rhythm pattern)
- **Report / Block tagger menu items are visible-but-no-op stubs.** ¬ß12 calls for both in the kebab; the phase board only spec'd Untag fully. Kept Report + Block as visible options that close the sheet so the menu reads complete; real moderation backend is post-demo. ‚Üí no feature-doc update needed
- **Per-dog Photos settings gear surfaces owner `tagApproval` read-only, NOT a per-dog override.** Per-dog override is V2+ (¬ß12). The gear's "Tag approval" row mirrors the owner's setting and links to `/profile?tab=about` to edit. ‚Üí `features/profiles.md` (updated)
