# Cross-Shelter Mentor Network — Walkthrough

Verification checklist for the Cross-Shelter Mentor Network phase. **Concise by design** — the build was driven end-to-end in the preview during development (mentee loop, graduation, solo walk booking, portability, non-accepting fallback, credit toggle all exercised); this walkthrough surfaces what's worth YOUR judgment.

**How to use:**

1. Run the dev server (`npm run dev`, port 3000).
2. Switch personas via the profile-page name dropdown, the `/` launcher, or the `?as=<personaId>` URL param.
3. **Most items want a fresh demo state** — clear localStorage (or use a private window) before V1; later V items build on V1's state deliberately.

**Status legend:** `[ ]` not yet walked · `[x]` walked, no issues. Multi-step flows are a label header + one checkbox per observable step; single checks keep the box on the header. **Keep V items as pass/fail observations — the *why* lives in "Decisions surfaced," not here.**

## Decisions surfaced during walkthrough

*(Append as you walk — each entry carries a `→ target-doc.md` annotation for the phase-close sweep.)*

1. **`booking_confirmation` chat artifact for direct-booked services** (O1 review, 2026-06-10). Direct booking ratified — but the chat keeps a structured booking artifact: a new `booking_confirmation` `MessageType` + `ChatMessage.bookingRef` + `BookingConfirmationCard` (reuses the `.inbox-proposal-*` accepted treatment; "Booking confirmed" header, no sign stamp, View booking → link). Appointment-confirmation framing, NOT contract framing — nothing to approve. The chronicle rule generalizes: *every booking in a relationship is findable from its chat thread*, whether it arrived via proposal-sign or direct booking. Generic shape — any future instant-book flow posts the same type. → `features/messaging.md` (new message type + card) + `strategy/Groups & Care Model.md` → Services as Catalog (mentor_session booking path).
2. **Completed mentor session → mutual Connected** (O1 review, 2026-06-10). The mentor flow has no contract-sign (the Connected trigger on other paid flows), so the first COMPLETED session marks mentor + mentee mutually Connected — supervised in-person dog handling is a stronger trust event than a signature, arguably the most on-thesis Connected transition in the app. Booking alone still only marks Familiar (consistent with inquiry-send). Implemented in `useMentorSessionCompletion` (idempotent across sessions). Alternative considered and parked: prompting both parties instead of auto-marking — revisit if user testing shows auto-Connected feels presumptuous. → `strategy/Trust & Connection Model.md` → "Inquiry-driven transitions" (add the mentor-session rule).
3. **Tomáš's persona copy carries the mentee arc** (O2 review, 2026-06-10). Landing card goal reworked: *"Boards Hugo with a sitter he trusts when work travel hits — and wishes his 9-to-6 left room to walk shelter dogs"* — the wish framing stays honest in fresh demo state (his mentee journey is drivable, not pre-staged) and sets up the mentor path as its answer. Switcher tagline gains "; drawn to shelter dogs." His demo-mode highlight reel should pick up the mentee surfaces (mentor booking, graduation, volunteer-work section) at phase close. → `features/demo-mode.md` (Tomáš highlight reel + cast description refresh).
4. **Tier thresholds are suggestions; the shelter promotes/demotes freely** (O4 review, 2026-06-10). The gap O4 exposed: tier escalation auto-promoted on walk count alone — 25 walks crowned a Super Volunteer with no shelter say, and that flowed into platform tier + mentor eligibility. Resolution (PO call): NOT an approval step (that recreates the admin queue the phase removes) — auto-promotion stays the zero-admin default, but the shelter holds free override authority both directions. Data: persisted `tierOverrides` map keyed `(shelterId, userId)` in `WalkerApplicationsContext`; effective tier = override ?? seeded/derived, threaded through every reader (affiliations, platform tier, mentor gate, Members tab, dog-page eligibility). Surface: "(demo)" dots dropdown per Members-tab walker row (operator stub per scope discipline 2; real home is FC16 walker pool management, updated). The shelter's lever reaches the whole stack: demoting a mentor's only trusted affiliation revokes platform Super Volunteer + mentor offerings (verified live). *(Initial "tier set by shelter" row marker removed same day — see #6.)* Rode along: static-roster walkers (Klára) now resolve as vouched on dog pages — they could not book walks before. → `features/shelters.md` (tier model: thresholds-as-suggestions + override authority + provenance marker) + `implementation/badges.md` (volunteer tier earning note).
5. **Members-row kebab is the per-walker operator surface** (O4 follow-up + O6 reshape, 2026-06-10; labels finalized same day). The dots trigger sits flush right (descendant-selector specificity bump to defeat the generic `.dropdown-menu-wrap { flex: 1 }` — P76 pattern); the menu opens leftward. Actions today, **no "(demo)" suffixes** (PO call): **Promote / Demote** act for real — their effect is immediately visible on the pill, so no label disclaimer needed; **Credit walks** and **Remove from walkers** (destructive slot, added on PO question — the shelter-level "block" folds into removal as a don't-readmit flag, distinct from person-to-person Block) fire the stub-feature toast (real flows need the operator surface: count + period form for crediting; reason + reapply policy + a removed-walkers layer for removal). 6. **Surfaces show facts; provenance stays in the data** (PO call, 2026-06-10). The walker subline briefly carried up to three clauses ("64 walks · 60 credited by the shelter · tier set by shelter") — auditing information nobody reading a roster needs. Resolution: public surfaces (Members rows, profile Volunteer-work rows) render the plain total + tier pill and TRUST them; the credited/logged split and override-vs-derived distinction remain DATA-layer facts (`creditedWalkCount`, `tierOverrides` — A7's audit trail intact) that surface on the future shelter admin view (FC16), where someone actually has the auditing job. Open thread parked: whether to offer shelters non-mentor verification alternatives (some won't adopt the Super-Volunteer-mentor model at all) — the direct-apply path + per-shelter `acceptsMentorVouches` opt-out already cover "no thanks"; richer in-between shapes wait on coordinator feedback. → `features/shelters.md` (mentor discovery surfaces) + `strategy/Groups & Care Model.md` → Services as Catalog (mentor_session entry points).
