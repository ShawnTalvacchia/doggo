# Cross-Shelter Mentor Network — Walkthrough

Verification checklist for the Cross-Shelter Mentor Network phase. **Concise by design** — the build was driven end-to-end in the preview during development (mentee loop, graduation, solo walk booking, portability, non-accepting fallback, credit toggle all exercised); this walkthrough surfaces what's worth YOUR judgment.

**How to use:**

1. Run the dev server (`npm run dev`, port 3000).
2. Switch personas via the profile-page name dropdown, the `/` launcher, or the `?as=<personaId>` URL param.
3. **Most items want a fresh demo state** — clear localStorage (or use a private window) before V1; later V items build on V1's state deliberately.

**Status legend:** `[ ]` not yet walked · `[x]` walked, no issues. Multi-step flows are a label header + one checkbox per observable step; single checks keep the box on the header. **Keep V items as pass/fail observations — the *why* lives in "Decisions surfaced," not here.**

Two seeded mentors: **Klára** (trainer — 450 Kč/session at Útulek + Pes v nouzi) and **Pavel D.** (senior volunteer, 87 walks — 350 Kč/session at Útulek only; the A5 "mentors don't have to be trainers" case). Klára is also seeded as Útulek's second trusted walker (64 walks · 60 shelter-credited, data-level provenance).

## Decisions surfaced during walkthrough

*(Append as you walk — each entry carries a `→ target-doc.md` annotation for the phase-close sweep.)*

1. **`booking_confirmation` chat artifact for direct-booked services** (O1 review, 2026-06-10). Direct booking ratified — but the chat keeps a structured booking artifact: a new `booking_confirmation` `MessageType` + `ChatMessage.bookingRef` + `BookingConfirmationCard` (reuses the `.inbox-proposal-*` accepted treatment; "Booking confirmed" header, no sign stamp, View booking → link). Appointment-confirmation framing, NOT contract framing — nothing to approve. The chronicle rule generalizes: *every booking in a relationship is findable from its chat thread*, whether it arrived via proposal-sign or direct booking. Generic shape — any future instant-book flow posts the same type. → `features/messaging.md` (new message type + card) + `strategy/Groups & Care Model.md` → Services as Catalog (mentor_session booking path).
2. **Completed mentor session → mutual Connected** (O1 review, 2026-06-10). The mentor flow has no contract-sign (the Connected trigger on other paid flows), so the first COMPLETED session marks mentor + mentee mutually Connected — supervised in-person dog handling is a stronger trust event than a signature, arguably the most on-thesis Connected transition in the app. Booking alone still only marks Familiar (consistent with inquiry-send). Implemented in `useMentorSessionCompletion` (idempotent across sessions). Alternative considered and parked: prompting both parties instead of auto-marking — revisit if user testing shows auto-Connected feels presumptuous. → `strategy/Trust & Connection Model.md` → "Inquiry-driven transitions" (add the mentor-session rule).
3. **Tomáš's persona copy carries the mentee arc** (O2 review, 2026-06-10). Landing card goal reworked: *"Boards Hugo with a sitter he trusts when work travel hits — and wishes his 9-to-6 left room to walk shelter dogs"* — the wish framing stays honest in fresh demo state (his mentee journey is drivable, not pre-staged) and sets up the mentor path as its answer. Switcher tagline gains "; drawn to shelter dogs." His demo-mode highlight reel should pick up the mentee surfaces (mentor booking, graduation, volunteer-work section) at phase close. → `features/demo-mode.md` (Tomáš highlight reel + cast description refresh).
4. **Tier thresholds are suggestions; the shelter promotes/demotes freely** (O4 review, 2026-06-10). The gap O4 exposed: tier escalation auto-promoted on walk count alone — 25 walks crowned a Super Volunteer with no shelter say, and that flowed into platform tier + mentor eligibility. Resolution (PO call): NOT an approval step (that recreates the admin queue the phase removes) — auto-promotion stays the zero-admin default, but the shelter holds free override authority both directions. Data: persisted `tierOverrides` map keyed `(shelterId, userId)` in `WalkerApplicationsContext`; effective tier = override ?? seeded/derived, threaded through every reader (affiliations, platform tier, mentor gate, Members tab, dog-page eligibility). Surface: "(demo)" dots dropdown per Members-tab walker row (operator stub per scope discipline 2; real home is FC16 walker pool management, updated). The shelter's lever reaches the whole stack: demoting a mentor's only trusted affiliation revokes platform Super Volunteer + mentor offerings (verified live). *(Initial "tier set by shelter" row marker removed same day — see #6.)* Rode along: static-roster walkers (Klára) now resolve as vouched on dog pages — they could not book walks before. → `features/shelters.md` (tier model: thresholds-as-suggestions + override authority + provenance marker) + `implementation/badges.md` (volunteer tier earning note).
5. **Members-row kebab is the per-walker operator surface** (O4 follow-up + O6 reshape, 2026-06-10; labels finalized same day). The dots trigger sits flush right (descendant-selector specificity bump to defeat the generic `.dropdown-menu-wrap { flex: 1 }` — P76 pattern); the menu opens leftward. Actions today, **no "(demo)" suffixes** (PO call): **Promote / Demote** act for real — their effect is immediately visible on the pill, so no label disclaimer needed; **Credit walks** and **Remove from walkers** (destructive slot, added on PO question — the shelter-level "block" folds into removal as a don't-readmit flag, distinct from person-to-person Block) fire the stub-feature toast (real flows need the operator surface: count + period form for crediting; reason + reapply policy + a removed-walkers layer for removal). Functional crediting remains only on the walker-button demo dropdown (bootstrap beats); credits on a static-roster walker still accrue on top of seeded history at the data level. The fuller candidate action set — remove from pool, pause walker, coordinator notes, per-walker dog restrictions, message-via-staff — is enumerated in FC16 for the real operator surface, not built. → `features/shelters.md` (Members tab: per-walker operator stub) + FC16 (done).
6. **Surfaces show facts; provenance stays in the data** (PO call, 2026-06-10). The walker subline briefly carried up to three clauses ("64 walks · 60 credited by the shelter · tier set by shelter") — auditing information nobody reading a roster needs. Resolution: public surfaces (Members rows, profile Volunteer-work rows) render the plain total + tier pill and TRUST them; the credited/logged split and override-vs-derived distinction remain DATA-layer facts (`creditedWalkCount`, `tierOverrides` — A7's audit trail intact) that surface on the future shelter admin view (FC16), where someone actually has the auditing job. If the shelter alters something, the displayed state simply changes — no annotation about how it got there. Considered and rejected along the way: a checkmark "shelter-approved" stamp (every rostered walker is shelter-approved — a check on some rows would misread the others as unapproved) and moving the tier pill to the right rail (breaks the pill-next-to-subject convention; truncates names on mobile). → `features/shelters.md` (Members tab rendering rule) + the shelter-demo doc Beats 1 + 5 (updated in place).
7. **Mentor discovery is list-first and shelter-neutral** (PO direction, 2026-06-11). A shelter surface must never feature a specific mentor — spotlighting one provider's paid offering reads as the shelter advertising a favorite (framing principle), and the old card was just `mentors[0]` from the single-mentor world. Shipped: the shelter mentor card is mentor-neutral ("from {min} Kč") with a **See mentors** CTA → `MentorListSheet` → booking sheet **locked to the shelter** (no shelter-switcher pills in shelter/dog context — the shelter was chosen by where the user tapped; the picker survives only on mentor-profile entry, where no shelter is chosen yet). Mid-mentorship the card routes straight to the mentee's own mentor at their price (mentorship begins at first booking; the list is never re-offered). Pavel D. seeded as a second Útulek mentor (350 Kč, senior volunteer — the A5 "mentors don't have to be trainers" case made visible + mentor-set price variety). Slots-calendar booking noted as the production direction (no slot model exists anywhere in the prototype — FC17). **List-sheet refinements (same-day PO review):** intro copy reframed to the mentee benefit ("Get certified to volunteer by walking with a mentor {min} times. Mentor walkers are experienced handlers who know {shelter}'s dogs and routines…") with no payment mechanics up top; **Super Volunteer badge dropped from the rows** (listing already requires the tier, so the badge is redundant); availability shown per-offering via a new `CarerMentorSessionServiceConfig.availabilityLabel` (narrow explicit daytime times — evenings removed in #8 — because mentoring is a side line of a side hustle, NOT the carer's full availability grid); each option is a **rounded card that lifts on hover**. Open thread parked: whether to offer shelters non-mentor verification alternatives (some won't adopt the Super-Volunteer-mentor model at all) — the direct-apply path + per-shelter `acceptsMentorVouches` opt-out already cover "no thanks"; richer in-between shapes wait on coordinator feedback. → `features/shelters.md` (mentor discovery surfaces) + `strategy/Groups & Care Model.md` → Services as Catalog (mentor_session entry points).
8. **Mentor booking sheet — visual progress + time-of-day + waiver links** (PO review, 2026-06-11). Four refinements to the mentor booking modal: **(a) Visual progress track** — `MentorProgressTrack` replaces the prose "after N sessions…" line on accepting shelters: numbered nodes (done = filled violet / current = violet ring / upcoming = grey) connected to a dashed 🎓 "Solo walker" goal node, with a short "Booking session X of N" caption. Renders only when the shelter accepts mentor-vouches (non-accepting + already-vouched keep prose). **(b) Time-of-day picker** — Morning / Afternoon segmented pills, **required** to submit; **no evening** (shelter walks run in daytime hours — Útulek's "weekdays 9–5"). The mentor confirms the exact time; their `availabilityLabel` shows as a hint. Persists to `Booking.ownerNotes` ("Preferred time: Morning") — no typed slot field, since the prototype has no slot model (slots-calendar is the FC17 production direction). **(c) Waiver names are links** — each waiver name is an inline `span[role=button]` (NOT `<button>` — its inline-block UA default orphaned the trailing period when the link wrapped) firing the stub-feature toast; the real version opens the document for review before signing. Checkbox label reads "I've read and agree to {link}." **(d) Checkbox alignment** — waiver rows top-align the box with the first line of a wrapping label (`.mentor-waiver-row`). **Dog picker — see #9 for the settled model:** the mentee is NOT locked to one dog (or mentor) across sessions; the dog is a transient entry hook only. (This bullet went through two same-day reversals; #9 is the final word.) **Evenings also removed from the seeded mentor availability** (Klára "Sat mornings · Wed afternoons"; Pavel "Sat afternoons · Sun mornings" — weekend slots reinforce the latent-demand thesis). → `features/shelters.md` (mentor booking sheet: progress track + time-of-day + waiver-link pattern) + FC17 (slots-calendar production direction) + `planning/Future Considerations.md` FC16 (real waiver-document review surface).
9. **Each mentored session is independent — no pinned mentor, no pinned dog** (PO direction, 2026-06-11; this is the SETTLED model after two corrections same day). The mentored path is a count of N sessions toward the SHELTER's vouch; each session can be a different mentor and a different dog — they just step the count. Two things that briefly over-coupled, now removed: **(a) Pinned mentor** — the mid-mentorship card used to re-book "your mentor." Now the card's "Book next session" CTA opens the mentor LIST every time (pick anyone); `completeMentorSession` attributes Connected + the graduation vouch to whichever mentor ran THAT session (the completed booking's carer), not a pinned one. **(b) Pinned dog ("working toward {dog}")** — briefly persisted on the mentorship + booking + card + progress line; REMOVED entirely. Locking one dog actually works against adoption: the point is sampling different dogs across sessions to find the one you'd adopt. The dog now appears ONLY as a transient ENTRY hook — the dog-page CTA ("Want to walk {dog}? Get certified with a mentor — {min} sessions and you walk solo here"; restricted: "{dog} needs an experienced handler — get certified with a mentor and build up") and the list intro ("Want to walk {dog}?") — which motivate without locking anything downstream. After vouching, the walker picks any eligible dog on the solo `WalkBookingSheet`. **Session numbering** counts committed bookings (completed + upcoming) so booking ahead doesn't repeat "session 1." → `features/shelters.md` (mentored path: independent sessions, entry-hook-only dog reference) + the shelter-demo doc (funnel: dog draws you in, certification is general, adoption follows vouching).
10. **Mid-mentorship card: stepper + violet CTA below it** (PO direction, 2026-06-11). The card flips to a vertical layout — headline ("N more sessions to walk solo") → full-width `MentorProgressTrack` (extracted to `components/shelters/MentorProgressTrack.tsx`, reused by the booking sheet) → CTA below it (was crammed to the right). The "Book next session" CTA uses a **volunteer-violet** treatment (`.mentor-violet-cta`, violet-700) — scoped to this surface, NOT a new global ButtonAction variant, so it doesn't add a colour to the app-wide button system; it applies the volunteer family (which already owns the stepper + credential pills + Volunteer badges) to its own action, unifying the journey. Dial back to outline/neutral if it ever reads as colour sprawl. → `implementation/design-system.md` (note the scoped violet CTA + the principle: a semantic family may own its CTA without becoming a global variant).
