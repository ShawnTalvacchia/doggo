## Decisions surfaced during walkthrough

Append-only log of decisions and emergent design changes from this phase. Each entry needs to land in its home doc at phase close.

- **Visibility filter UI redesign — tag-type pill-dropdowns + multi-select via checkboxes, popover menu not modal.** The V1 design (one auto-pill per group the user had posted to, ModalSheet single-select) created a noisy auto-generated row + the wrong interaction shape. Replaced 2026-06-04 with fixed tag-type pills (`Dog ▾`, `Person ▾`, `Place ▾`, `Community ▾`, `Meet ▾`) opening `.dropdown-menu` popovers with checkbox rows. Multi-select within type (OR), AND across types. Albums-as-saved-filter-sets becomes the natural V2 forward direction (§12 open item — "albums-as-tag-type"). → `features/profiles.md`
- **Composer affordance moves to page-action slot.** ShareMomentBar in-panel was clunky alongside the new filter row. Retired from PostsTab; "Post" button (outline + CameraPlus icon) registers via `setPageAction` for the Posts tab — mirrors the Communities pattern. AppNav Camera+ stays suppressed via `setPageAction(node, { suppressCreate: true })` so they don't double up. → `features/profiles.md` (updated)
- **Highlights field renamed — `PetProfile.photoGallery` → `PetProfile.highlights`.** Initial scope kept the legacy name to save churn; reversed 2026-06-04 after Highlights also landed on `UserProfile`. The asymmetric naming (`photoGallery` for dogs, `highlights` for users) was confusing enough to be worth the mechanical rename. ~6 callsites updated + seed migration. → `features/profiles.md` (updated)
- **Highlights = a tag/state on a photo, not a separate collection.** A first pass of the Edit Highlights modal added an inline "Add photos" picker (grid of unpinned album photos with `+` overlays). Removed 2026-06-04 — adding to Highlights is an in-context action via the per-post three-dot menu. Conceptual model: Highlights is a property of a photo (pin/unpin), not a list the owner builds. The Edit modal stays for order + bulk unpin only. Aligns with the §12 V2 direction (albums-as-tag-type). → `features/profiles.md` (updated)
- **Global post-detail surface — `PostLightbox`, no `/posts/[id]` route.** Surfaced as a need when wiring the tag_pending notification (no way to see the parent post) AND when the user asked for a full-screen image experience. Built `contexts/PostDetailContext.tsx` — `usePostDetail().openPost(postId, { collection? })` overlays the app with a photo-led lightbox: dark backdrop, photo dominant on left (~60% desktop, full-bleed-top on mobile), sidebar on right (or below on mobile) with author + caption + tags + reactions + comments + PostKebabMenu. When opened with a `collection`, ← / → keys and on-screen prev/next traverse the album in lightbox mode; without a collection (e.g. from a tag_pending notification), single post with no cross-post nav. Multi-photo posts get sub-photo navigation + `N/M` counter. No `/posts/[id]` route — comments aren't a primary platform thread for Doggo; the lightbox is sufficient and keeps "tap → see → close" consistent. PhotoGrid passes the visible posts as the collection so the lightbox can navigate the album. → `features/profiles.md` (note as the canonical post-detail surface)
- **Stub actions now open a "Demo placeholder" toast.** Previously stub menu items just closed the sheet — silently inert. Replaced 2026-06-04 with a shared `StubFeatureContext` (`contexts/StubFeatureContext.tsx`) that slides in a non-modal toast (top-right desktop / top mobile, ~6s auto-dismiss, × to dismiss) identifying the feature and what it's waiting on. Built on top of a new reusable `<Alert>` component with 4 variants (info / success / warning / error) — also available for inline page-level callouts. Wired into PostKebabMenu's four stubs (Edit post / Delete post / Report / Block). Available app-wide via `useStubNotice()` for future stub actions. → `implementation/design-system.md` (note as a stub-action pattern)
- **Kebab is a floating popover, not a modal.** Initial implementation wrapped the per-post kebab menu in a `ModalSheet` (full-screen sheet). Replaced 2026-06-04 with a `.dropdown-menu` popover anchored top-right of the kebab button, click-outside-to-close, right-aligned with `min-width: 200px / max-width: 280px / width: max-content` so labels never wrap. Reuses the standard project pattern (same as Follow / RSVP / Sort menus). Lighter feel for a short list of actions; works on mobile + desktop. Specificity-fix note in punch list — the shared `.dropdown-menu` rule comes later in source order so variant overrides need double-class selectors (`.dropdown-menu.post-kebab-menu`). → no feature-doc update needed
- **Per-post kebab is the unified action surface; long-press removed.** The earlier long-press menu on auto-album tiles (Pin / Hide) read as finicky. Replaced 2026-06-04 with an expanded `PostKebabMenu` that covers all per-post actions: Pin/Unpin to Highlights, Hide/Show in album, Edit post (caption + tags consolidated into one stub), Delete post; plus the existing Untag {Dog} / Report / Block {Author} for non-owner posts. **Context-aware Highlights destination:** menu reads `usePathname()` — on `/dogs/[id]` for a viewer-owned dog, Pin targets that dog's Highlights and Hide-from-album becomes available; on any other surface (feed, owner profile), Pin targets the viewer's own Highlights and Hide is suppressed (no album in context). Replaces the previous one-row-per-destination model with a single state-toggling row. `AlbumTileLongPressMenu` deleted. → `features/profiles.md` (updated)
- **"posts ≈ photos" — no dedicated Photos tab.** Posts and Photos are the same collection (every Post has 1–4 photos). Posts tab gets a List ⇄ Grid view toggle instead of a parallel Photos tab. → `features/profiles.md` (updated)
- **Highlights strip lives on `/dogs/[id]`, NOT on profile-level PetSummaryCard.** Reinforces Dog Profile phase's "canonical dog page" thesis. Owner-level "all my dogs" aggregate explicitly NOT built. → `features/profiles.md` (updated)
- **Empty-state composer CTA opens the picker without pre-tagging the specific dog.** `openComposer({ initialTagPicker: "dog" })` opens the dog tag picker — pre-selecting THIS specific dog needs a `prefilledTags` field on `PostComposerContext` which is out of scope. One-tap fill from the picker is acceptable for V1. → no feature-doc update needed
- **Untag is a per-viewer suppression filter, not a post mutation.** Prototype has no editable-post context, so "Untag Bára" persists per viewer via `useUntagStore` and feeds into `/dogs/[id]` auto-album filtering. The post's `Post.tags[]` itself is unchanged — the dog tag chip still renders on post chrome until a real editable-post store lands. → `features/profiles.md` (updated)
- **Pending-tag approval model: grandfathered-default + explicit `DEMO_PENDING_TAGS` carve-out.** Initial implementation used a 14-day cutoff on both queue + album, which produced auto-approval-by-aging — semantically wrong. Replaced 2026-06-04: dog-tag-on-owned-dog pairs default to approved (grandfathered); pairs in `DEMO_PENDING_TAGS` (lib/usePendingTagsStore.ts) default to pending. Pending stays pending forever until owner approves or rejects. Demo curation is a one-line edit to the constant. → `features/profiles.md` (updated)
- **Pending-tag approval surface = notifications, NOT a dedicated About-tab section.** Initial implementation rendered `PendingTagsSection` inline on `/profile?tab=about`. Matches no shipping social platform pattern (Instagram / Twitter / Facebook all use notifications for tag review). Refactored 2026-06-04: added `tag_pending` to `NotificationType`, seeded 2 entries in `mockNotifications` matching `DEMO_PENDING_TAGS`, added inline Approve / Reject footer to the notifications row (mirrors `connection_request`). Resolved notifications filter out via `usePendingTagsStore.getDecision`. `PendingTagsSection` deleted. → `features/profiles.md` (updated)
- **Photos/Posts unified across owner profile + dog page.** Initial implementation diverged: dog page had separate "Photos" grid + "Posts about {Dog}" sections; owner profile had a single Posts tab with List⇄Grid toggle + tag-type filters. Unified 2026-06-04 around the owner-profile shape: new `components/posts/PostsCollectionView.tsx` carries filter pills + view toggle + list/grid render. Used by both `PostsTab` AND the dog page (which drops the separate Photos + Posts-about sections in favor of a single Posts section under Highlights). Dog page section header is "Posts" (was "Posts about {Dog}"). Highlights strip + Edit modal + `usePhotoAlbumOverrides` generalized from dog-only to subject-id (dog OR user). New `UserProfile.highlights?: string[]` field seeded for the four anchor personas. Forward direction logged: Highlights becomes one album in a dropdown switcher when §12's "albums-as-tag-type" V2 lands. → `features/profiles.md` (updated)
- **Resolver split — `getPostsByDog` (viewer-aware, `lib/dogPosts.ts`) vs `getDogPosts` (ungated, `lib/mockShelters.ts`).** The shelter-side ungated helper survives for `RecentWalkers` (where walker activity is a public-ish surface; gating would over-restrict). Auto-album surfaces use the new helper. → no feature-doc update needed (architectural)
- **Right-click on album tile triggers the long-press menu (desktop convenience).** Touch gets press-and-hold; mouse users get right-click as the equivalent affordance. Click-handling is intercepted so a long-press doesn't ALSO open the post modal. → no feature-doc update needed
- **Punch-list spawn: P75 — panel chrome glassmorphism consistency.** Some PageColumn panels glass, most don't. Recommendation logged: go opaque app-wide. Not in scope for this phase. → `phases/punch-list.md`
- **Section rhythm — global tight-header + larger body-gap pattern.** Two parallel section surfaces (`.profile-tab-stack > section` and `.dog-profile-section`) had diverged on internal gap (4px vs 16px). Aligned 2026-06-04 to a single rhythm rule that both classes share (visual chrome — padding/background/border — stays per-class because they render in different layout contexts): `gap: var(--space-sm)` (tight header → first body item) + `> :nth-child(n+3) { margin-top: calc(--space-md - --space-sm) }` (larger gap between body items). CSS-only via a comma-separated selector list — no JSX wrappers required. Future section surfaces can join by adding to the list. Also tightened `.dog-profile-section--header-only` padding-bottom (Posts about {Dog}) from `--space-lg` → `--space-sm` so the feed-card's own top padding doesn't stack with the section's bottom padding into an oversized gap. → `implementation/design-system.md` (note as a recurring rhythm pattern)
- **Report / Block tagger menu items are visible-but-no-op stubs.** §12 calls for both in the kebab; the phase board only spec'd Untag fully. Kept Report + Block as visible options that close the sheet so the menu reads complete; real moderation backend is post-demo. → no feature-doc update needed
- **Per-dog Photos settings gear surfaces owner `tagApproval` read-only, NOT a per-dog override.** Per-dog override is V2+ (§12). The gear's "Tag approval" row mirrors the owner's setting and links to `/profile?tab=about` to edit. → `features/profiles.md` (updated)
- **Filter menu popover sized within 220–340px width with `text-overflow: ellipsis`.** Long community names like "Klára's Calm Dog Sessions" render on one line; truncate before wrapping. Surfaced during the B-redesign verification — fixed in the same session. → no feature-doc update needed
- **Filters: explicit "+ Filter" pattern, NOT auto-rendered empty type pills.** First version (post the modal-to-popover refactor) rendered one pill per filter type permanently — `Dog ▾ Person ▾ Place ▾ Community ▾ Meet ▾`. Read as noisy chrome the user had to learn was "filters" and crowded narrow viewports. Replaced 2026-06-04 with an explicit `+ Filter` button (dashed-border chip) that opens a menu of available filter types; picking a type sets `pendingType` and renders that type's pill in `defaultOpen` mode so the user lands directly in the values picker. Active filter pills wrap to a SECOND row below the top row (which anchors +Filter left, view toggle right) — top row layout stays stable regardless of how many filters are active. Pills + toggle matched to 32px height for a clean siblings-on-one-row read. Sets up §12 V2 "albums as saved filter sets" naturally — albums slot in as another pill type. → `features/profiles.md` (updated)
- **PostKebabMenu: Pin / Hide available regardless of authorship.** First version gated Pin and Hide behind `isAuthor`, on the assumption that you only feature/curate your own photos. Wrong call — Highlights and the per-dog album are about the VIEWER's surfaces, not about who took the photo. A walker's post of your dog is exactly the kind of photo you'd want pinned. Refactored 2026-06-04 into three layers: curation actions (Pin/Hide) universal and gated only by surface context (dog-page context = ownership-gated via `useDogContext`); Edit/Delete author-only; Untag/Report/Block non-author-only. → `features/profiles.md` (updated)
- **Highlights URLs resolved globally; lightbox carousel = the highlights themselves.** First wire scanned only the subject's posts to resolve a highlight URL back to a post. Broke for highlights pinned from another author's post (e.g., Tereza pinning Eva's photo of Bella) — those thumbs fell back to non-clickable divs. Switched 2026-06-04 to **option A** from the original framing: each highlight URL resolves against `mockPosts` globally, and the lightbox carousel scopes to the **Highlights themselves** (each resolved to its source post), not to the subject's posts collection. ← / → moves through highlights in order. Generalizes cleanly across cases where highlights span authors — the curated set IS the collection. → `features/profiles.md` (updated)
- **Lightbox: within-post nav hidden in Highlights mode + per-item `photoIndices`.** Two coupled changes: `openPost(postId, opts)` gained `withinPostNav?: boolean` (default true) which hides the small `< >` photo-edge arrows + `N/M` counter, and `photoIndices?: number[]` (parallel to `collection`) so each cross-post navigation step seeds the new post's photo index from a per-entry value (not always 0). HighlightsStrip passes both: within-post nav hidden because each highlight is a single curated photo (not a way to browse the post's other photos), and photoIndices ensures ← / → between highlights lands on each highlight's specific photo. Regular grid/list taps keep within-post nav enabled. → `features/profiles.md` (note as part of the post-detail lightbox spec)
- **PostLightbox mobile: full-document scroll + edge-attached chevron hints + swipe nav.** Switched the mobile layout from "locked photo + tiny sidebar scroll" to "the whole overlay scrolls as one document" — photo at top (55dvh), caption + comments flow below, overlay handles scroll. Trade-off: scrolling pushes the photo (and cross-post chrome) out of view; user can scroll back up for the carousel. Close × pins via `position: fixed` so it's always reachable. Cross-post nav stays visible on mobile as **subtle edge-attached chevron hints** (slim 28×56 chevrons at low opacity on photo edges) — discover the swipe affordance for users who don't think to try it, remain tappable. Touch swipe ←/→ on the photo area triggers cross-post nav (48px threshold + axis-dominance check so vertical scrolls don't trigger). FC14 logs the Instagram-style drag-over-photo sheet as the eventual polish step. → no feature-doc update needed (post-detail lightbox covered in profiles.md)
- **Tag-pending notification: preview thumbnail + "Approve to add to {Dog}'s album." copy.** Two coupled improvements to the V5 tag-approval flow. Thumbnail on the right of each tag_pending row (~56px square) renders the post's first photo so the owner can see what they're approving without tapping in — matches the Instagram / Facebook tag-review pattern. Body copy switched from caption excerpt to "Approve to add to Bára's album." which names the action's effect directly. Reflects the **Instagram tag model** we adopted: the tag exists in `Post.tags[]` from the moment the tagger publishes (visible to others); approval gates ONLY the dog's album surface. The earlier body copy left users wondering what "approve" did when they could already see the post + tag chip via the lightbox. Timestamp moved to a right column above the thumbnail so it remains the top-right element (consistent with all other notification rows). → `features/profiles.md` (updated)
- **Punch-list spawn: P76 — `.dropdown-menu` source-order specificity.** Variants of `.dropdown-menu` (e.g., right-aligned kebab menu sizing) lose to the base rules because base comes later in `globals.css`. Workaround: double-class selectors. Better long-term: move base into a CSS layer. Filed during the kebab popover refactor. → `phases/punch-list.md`
- **Punch-list spawn: `.dropdown-menu-wrap { flex: 1 }` class collision.** The wrap class makes pill+menu containers grow to equal flex share, which fights sibling layout. Hit twice in this phase — once on the filter pills (auto-gap between active filters), once on the lightbox sidebar header (kebab landing in the center instead of the far right). Both fixed with scoped overrides. Same root cause as P76 (source order) — extending P76 to cover the wrap class too. → `phases/punch-list.md` (extend P76)
